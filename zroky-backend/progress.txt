в”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђ
Scope (locked):
  - Idempotent replay dispatch endpoint suitable for the
    `zroky/regression-ci@v1` GitHub Action.
  - Default golden set per project (migration 0060) so the Action can
    POST without naming a set.
  - `summary_url` returned on every dispatch so the Action can post a
    PR-check `details_url` pointing at the dashboard.

Decisions (with reasoning):

  - Idempotency key = (project_id, golden_set_id, git_sha). NOT a
    client-supplied Idempotency-Key header. The Action retries on
    network failures and re-runs of the same workflow run, both of
    which share the same git SHA; a server-side key removes one
    failure mode (mis-implemented Idempotency-Key generation in the
    Action) and matches the natural "one grade per commit-against-set"
    cardinality.

  - Idempotency horizon split by run status:
        * non-terminal (pending/running): ALWAYS reuses (no time bound)
        * terminal     (pass/fail/error): reuses only if <60 min old
    Older terminal runs are intentionally NOT reused вЂ” a customer
    re-running the same commit after fixing a judge model or
    upgrading the model gets a fresh row. 60-min default lives in
    `IDEMPOTENCY_TERMINAL_HORIZON_MINUTES`.

  - Manual triggers (git_sha=None) are NEVER deduped. A human
    pressing "Run" twice means they want two runs. Whitespace-only
    git_sha values are normalized to None so " " can't masquerade as
    a real key.

  - branch_name / pr_number / commit_message stored in summary_json,
    NOT as new columns. Cardinality is tied to git_sha (already
    indexed), they're display-only, and the dashboard reads
    summary_json anyway. Truncated: branch_name 255 / commit_message
    200 (first line only) to keep the row bounded.

  - New `POST /v1/replay/dispatch` is its own router file
    (`app/api/routes/replay_dispatch.py`) вЂ” not folded into
    `replay_runs.py` because the surfaces have different auth
    expectations long-term (Action surface may grow signed-payload
    verification later). Default trigger here is "github" vs.
    "manual" on the goldens-router endpoint.

  - Default-set resolution priority:
        1) explicit golden_set_id in body
        2) projects.default_golden_set_id (new column, migration 0060)
        3) 422 with diagnostic message
    No "first golden set wins" fallback вЂ” silently picking would
    cause cross-set replays. Forces explicit configuration.

  - No FK on projects.default_golden_set_id в†’ golden_sets.id, matching
    the migration 0049/0050/0052/0053/0054 convention (RLS on
    golden_sets.project_id is the actual tenant boundary). Service
    layer cascades the clear: delete_golden_set() now zeroes any
    Project.default_golden_set_id pointing at it within the same
    project. Dangling pointers from race conditions map to 422 with
    "update the project's default_golden_set_id".

  - was_idempotent_hit() stamps a transient `_zroky_was_new` attribute
    on the returned ReplayRun rather than changing the return type
    of dispatch_replay_run. Keeps the existing service signature and
    caller code unchanged; route reads the flag to populate the new
    response field `idempotent: bool`.

  - Idempotent hits skip the Celery enqueue. The original dispatch
    already enqueued (or the run already finished). Re-enqueueing
    would race with the existing worker and waste a slot.

  - build_summary_url() reads FRONTEND_URL at call time (not
    import time). Test settings overrides work without
    cache_clear(). Falls back to https://zroky.com when unset.

Module 11 -- Weekly digest pipeline (compute + render + send)  (2026-05-14)
----------------------------------------------------------------

Scope (locked, plan section 10.5):
  - Promote weekly_impact.py -> digest_engine.py (M11.1 was done
    in Module 4.4 read-side; this module ships the WRITE side).
  - Audience-aware summary: engineer / manager / executive.
  - Pre-rendered HTML + on-the-fly plain-text fallback.
  - Two-stage Celery split (generate, then send) so a send-side
    SMTP outage cannot strand digest generation.
  - Admin trigger route for ops-driven regen.

Decisions (with reasoning):

  - AUDIENCE VOCAB COLOCATED. `digest_engine.AUDIENCES` and
    `billing_plans.DIGEST_AUDIENCE_VALUES` are the same vocabulary
    seen from two sides. A startup-time assertion
    (`_check_audience_vocab_in_sync`) catches drift in CI so a
    rename in one place crashes loudly instead of silently
    serving the wrong tier.

  - AUDIENCE RESOLVER FAIL-OPEN. `resolve_audience` returns
    DEFAULT_AUDIENCE ('engineer') on resolver failure or unknown
    value. Legacy weekly_impact did not consult entitlements at
    all, so degrading to 'engineer' is strictly better than
    skipping the project on a transient entitlement outage.

  - SUMMARY SHAPE = AUDIENCE FILTER. compute_summary RETURNS the
    blocks the audience is entitled to see -- engineer omits
    `pilot`/`replay`/`trend`; manager omits `trend`; executive
    omits `trend` when no prior digest row exists (no misleading
    "+0%" tile for brand-new projects). Renderers re-check the
    audience key as a defence-in-depth seam.

  - PREVENTED-WASTE PROXY. cost.prevented_waste_usd = sum of
    cost_total over failed calls whose error_code matches the
    fingerprint of an anomaly touched by an applied/reverted
    PilotAction in the window. Approximation (Anomaly has no
    direct call_id link in 0051) but the proxy is intentional:
    better to undercount prevented waste than invent a number.

  - TIER-2 PR URL SANITISER. Dry-run / recording sentinels
    (`dry-run://...`, `recording://...`) are filtered from the
    manager-tier `tier2_pr_urls` list. They are not real PRs the
    manager can review and surfacing them would erode trust.

  - REPLAY PASS-RATE DENOMINATOR DISCIPLINE. Replay runs whose
    summary_json lacks `trace_count_at_dispatch` are EXCLUDED
    from the rate calc, not counted as failures. Unknown
    denominator -> unknown rate (returns None), not 0.

  - WoW TREND OMITS BLOCK ON NO PRIOR. trend section is None for
    brand-new projects so the executive email does not render a
    confusing "+0%" tile for week 1. Renderer omits the section
    entirely.

  - HTML RENDERER IS ESCAPE-FIRST. recommendation + all dynamic
    strings pass through `html_escape` even though they are
    server-built today -- if a future change ever sources from
    user input the boundary already exists.

  - UPSERT PRESERVES sent_at. generate_weekly_digest UPDATES the
    existing row on regen (matched by project_id + week_start)
    and INTENTIONALLY DOES NOT TOUCH sent_at or sent_to_emails.
    A regen after the email task ran must not reset sent_at --
    otherwise the next send beat would email the same digest
    twice. Test: test_regen_preserves_sent_at.

  - TWO-STAGE CRON SPLIT. generate_weekly_digests (Mon 02:00 UTC)
    and send_pending_digests (every 5 min) are SEPARATE Celery
    tasks. An SMTP outage strands rows with sent_at=NULL; the
    next beat tick retries automatically. Conflating them would
    couple compute-side failures to send-side failures.

  - send_pending_digests SKIPS ROWS WITH NO RECIPIENTS. A project
    with no active admin/owner gets the row left untouched
    (sent_at stays NULL) -- the row simply waits for an admin
    to be invited. Logged via `digest_send_skipped_no_recipients`.

  - send_email RETURNING FALSE DOES NOT STAMP. Transient SMTP
    failure means sent_at stays NULL so the row retries on the
    next beat. Stamped only on send_email -> True.

  - ADMIN TRIGGER LIVES AT /internal (NOT /v1/internal). Plan
    section 10.5 names the route /v1/internal/digests/generate;
    the actual mount lives under the existing `/internal` router
    prefix to stay co-located with /internal/owner/* admin
    endpoints. Same auth seam (require_provisioning_access).

  - ADMIN TRIGGER DOES NOT SEND. The route only regenerates --
    operators who want an immediate send run send_pending_digests
    separately. Conflating "rebuild this week's digest" with
    "email it now" is the kind of footgun that loses customer
    trust on day one.

  - DIGEST_ENABLED KILL-SWITCH. Both Celery tasks short-circuit
    with `{"skipped": True, "reason": "DIGEST_ENABLED=false"}`
    when the setting is False. Lets ops disable the pipeline
    without touching beat schedule or revoking the binding.

Scope (locked, plan section 11.4):
  - Pull-driven sweepers for the two non-event transitions Stripe
    will never tell us about: no-card trial expiry and 7-day past-
    due grace expiry. Hourly Celery beat ticks, offset 7 minutes.
  - subscriptions.sla_tier ('none' | 'team' | 'enterprise') as a
    persistent column so reliability-SLA refund eligibility
    survives auto-downgrade and is queryable from /v1/billing/me.
  - FEATURE_LEGACY_BILLING flipped to False; legacy /v1/billing/
    {plans,subscription,usage} routes split into a separate
    legacy_router gated by the flag while the В§11.3 surface
    (checkout/portal/webhook/me) is always mounted.

Decisions (with reasoning):

  - TRIAL EXPIRY ELIGIBILITY = stripe_sub_id IS NULL.
    Application-managed (no-card) trials only. Paid-trial rows
    transition through Stripe's customer.subscription.updated;
    sweeping them here would race the webhook AND the
    _is_stale_event guard in stripe_sync would then drop the
    legitimate post-trial upgrade. Test
    test_skips_paid_trial_with_stripe_sub_id pins this.

  - PAST-DUE GRACE CLEARS stripe_sub_id ON DOWNGRADE.
    Customers with a paid-then-failed subscription have a real
    stripe_sub_id. After the 7-day grace cap we null it so a
    delayed invoice.paid for the dead subscription cannot
    resurrect Pro entitlements (invoice handler looks up by
    stripe_sub_id; missing row -> no-op). stripe_customer_id is
    retained for support and re-checkout convenience.

  - sla_tier IS PRESERVED ACROSS AUTO-DOWNGRADE.
    A customer who was on Team during the missed-SLA window
    keeps sla_tier='team' even after the row flips to
    (free, active). Refund-eligibility audits then remain
    reconstructable from a single row. Two tests pin this
