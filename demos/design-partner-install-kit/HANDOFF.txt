# Zroky Design-Partner Install Kit

This kit gives a design partner two runnable proofs that Zroky can protect an
autonomous agent action end to end:

1. Capture a high-stakes action from an agent trace.
2. Stop the unsafe action at runtime with a policy decision.
3. Configure and health-check the system-of-record connector.
4. Reconcile the agent's claimed result against that connector.
5. Export a redacted evidence pack with a stable evidence hash.

The kit is intentionally narrow. It proves the paid-launch loop for one
money-touching action and one customer-record action before expanding to DevOps,
access, procurement, email, or ticketing connectors.

## What The Partner Should Believe

A successful run means:

- Zroky saw the action before it committed.
- Zroky blocked or held the risky action instead of silently allowing it.
- Zroky configured the system-of-record connector and marked it healthy only
  after a saved connector test matched ground truth.
- Zroky marked the connector ready only after config, token, matched test,
  HTTP 2xx, and real read-attempt checks passed.
- Zroky checked the real ledger or CRM outcome, not just the agent's text output.
- Zroky produced an audit-ready evidence hash.
- No API key, ledger token, or CRM token was printed into the handoff artifacts.

It does not mean the whole deployment is production-ready. Live launch still
requires production secrets, billing/webhook smoke, worker readiness, and
gateway recovery proof.

## Local Proof

Run the refund proof first before touching partner systems:

```powershell
python scripts/run_design_partner_install_kit.py --scenario refund --json --write-summary artifacts/design-partner-refund-summary.json --write-evidence artifacts/design-partner-refund-evidence.json
```

Then run the CRM/customer-record proof:

```powershell
python scripts/run_design_partner_install_kit.py --scenario customer-record --json --write-summary artifacts/design-partner-crm-summary.json --write-evidence artifacts/design-partner-crm-evidence.json
```

Expected summary:

- `mode`: `local_demo`
- `runtime_policy.allowed`: `false`
- `connector_setup.status_endpoint`: `/v1/integrations/system-of-record/.../status`
- `connector_setup.config_endpoint`: `/v1/integrations/system-of-record/.../config`
- `connector_setup.test_endpoint`: `/v1/integrations/system-of-record/.../test`
- `connector_setup.health_status`: `healthy`
- `connector_setup.readiness_status`: `ready`
- `connector_setup.last_attempts`: `1` or higher
- `connector_setup.last_error_code`: `null`
- `connector_setup.last_retryable`: `null`
- `connector_setup.readiness_blockers`: `[]`
- `outcome_reconciliation.verdict`: `matched`
- `outcome_reconciliation.connector_attempts`: `1` or higher
- `outcome_reconciliation.connector_error_code`: `null`
- `evidence_pack.verification_status`: `pass`
- `proof.secrets_redacted`: `true`
- `evidence_pack.evidence_hash`: 64 hex characters

Artifacts:

- `artifacts/design-partner-refund-summary.json`: redacted refund proof summary.
- `artifacts/design-partner-refund-evidence.json`: redacted refund evidence pack.
- `artifacts/design-partner-crm-summary.json`: redacted CRM proof summary.
- `artifacts/design-partner-crm-evidence.json`: redacted CRM evidence pack.

## Connector Config Templates

Use these templates to map the partner's system of record before the first live
smoke. Replace placeholder values only; do not paste real tokens into tickets,
screenshots, or shared docs.

- `demos/design-partner-install-kit/ledger_refund_connector_config.example.json`
- `demos/design-partner-install-kit/customer_record_connector_config.example.json`

Each template contains:

- status endpoint: confirms whether Zroky has a saved connector.
- config endpoint: saves the base URL, path template, record path, and token.
- test endpoint: queries one safe existing record and reconciles it against the
  claimed fields.
- readiness contract: documents the required inputs, record fields, and checks
  that must pass before the connector is considered ready.
- pass criteria: `health_status=healthy`, `readiness_status=ready`,
  `verdict=matched`, attempts present, no `last_error_code`, and no leaked
  secret.

## Live Connector Preflight

Before running the full buyer proof, run the sandbox connector test endpoint by
itself. This catches bad tokens, wrong path templates, missing records, and
system-of-record shape mismatches before the runtime/evidence demo.

Refund preflight:

```powershell
python scripts/run_design_partner_install_kit.py --scenario refund --preflight-only --api-base-url https://api.zroky.ai --api-key <zroky_api_key> --ledger-base-url https://ledger.example.com/api --ledger-bearer-token <ledger_token> --refund-id <refund_id> --json --write-summary artifacts/design-partner-refund-preflight-summary.json
```

CRM/customer-record preflight:

```powershell
python scripts/run_design_partner_install_kit.py --scenario customer-record --preflight-only --api-base-url https://api.zroky.ai --api-key <zroky_api_key> --crm-base-url https://crm.example.com/api --crm-bearer-token <crm_token> --customer-id <customer_id> --json --write-summary artifacts/design-partner-crm-preflight-summary.json
```

Expected preflight summary:

- `connector_setup.config_saved`: `true`
- `connector_setup.health_status`: `healthy`
- `connector_setup.readiness_status`: `ready`

- `connector_setup.last_retryable`: `null`
- `connector_setup.readiness_blockers`: `[]`

## Live Partner Smoke

Use this after the partner has a Zroky project, API credential, and a ledger API
record that can be safely queried:

```powershell
python scripts/run_design_partner_install_kit.py --scenario refund --api-base-url https://api.zroky.ai --api-key <zroky_api_key> --ledger-base-url https://ledger.example.com/api --ledger-bearer-token <ledger_token> --refund-id <refund_id> --json --write-summary artifacts/design-partner-refund-live-summary.json --write-evidence artifacts/design-partner-refund-live-evidence.json
```

Use this after the partner has a CRM/customer API record that can be safely
queried:

```powershell
python scripts/run_design_partner_install_kit.py --scenario customer-record --api-base-url https://api.zroky.ai --api-key <zroky_api_key> --crm-base-url https://crm.example.com/api --crm-bearer-token <crm_token> --customer-id <customer_id> --json --write-summary artifacts/design-partner-crm-live-summary.json --write-evidence artifacts/design-partner-crm-live-evidence.json
```

Use `--bearer-token` instead of `--api-key` when the partner authenticates with
a user/session bearer token. Use `--project-id` when the API key is not already
bound to a single project.

## Required Inputs

| Input | Required for local | Required for live | Notes |
| --- | --- | --- | --- |
| `--api-base-url` | No | Yes | Zroky API base URL. Omit for local demo. |
| `--api-key` or `--bearer-token` | No | Yes | Zroky auth. Never paste into screenshots. |
| `--preflight-only` | No | Optional | Runs connector config + sandbox test only; no evidence pack. |
| `--ledger-base-url` | No | Yes | HTTPS ledger API base URL. |
| `--ledger-bearer-token` | No | Yes | Ledger auth token. Redacted from artifacts. |
| `--refund-id` | No | Yes | Existing refund id to verify in the ledger. |
| `--crm-base-url` | No | Yes for CRM | HTTPS CRM/customer API base URL. |
| `--crm-bearer-token` | No | Yes for CRM | CRM auth token. Redacted from artifacts. |
| `--customer-id` | No | Yes for CRM | Existing customer id to verify in CRM. |
| `--amount-usd` | No | Optional | Use when the partner refund amount differs from the fixture. |
| `--currency` | No | Optional | Defaults to the fixture currency. |
| `--status` | No | Optional | Defaults to the fixture status. |
| `--email`, `--account-id`, `--customer-status` | No | Optional | Use when CRM claimed fields differ from the fixture. |

## Pass Criteria

The handoff is a pass only when all of these are true:

- `captured_call_linked`
- `unsafe_action_stopped`
- `connector_configured`
- `connector_health_verified`
- `real_connector_ready`
- `matched_outcome_shown`
- `evidence_hash_visible`
- `evidence_pack_passed`
- `secrets_redacted`

Any false value blocks the pilot handoff. Do not call a run verified when the
status is `not_verified`, the outcome is `mismatched`, or the evidence hash is
missing.

## Failure Meanings

| Failure | Meaning | First fix |
| --- | --- | --- |
| `unsafe_action_stopped=false` | Runtime policy did not block/hold the risky action. | Check mandate, action type, impact amount, and policy thresholds. |
| `connector_configured=false` | The saved connector config was not accepted or was not active. | Check connector base URL, path template, token, project id, and admin role. |
| `connector_health_verified=false` | Connector setup exists but the latest saved connector test is not healthy/matched. | Open `connector_setup.last_error_code`, `last_error`, HTTP status, attempts, and the connector test check. |
| `real_connector_ready=false` | Connector health looked green but the readiness contract did not pass. | Open `connector_setup.readiness_blockers` and fix missing token, missing HTTP 2xx, missing attempts, or connector error code. |
| `matched_outcome_shown=false` | System of record did not match the agent's claim. | Compare refund id, amount, currency, and status. |
| `evidence_pack_passed=false` | Evidence exists but is not verified. | Open the evidence pack and inspect decision/outcome status. |
| `secrets_redacted=false` | A secret leaked into summary or evidence. | Stop the handoff, rotate the leaked token, and fix redaction before rerun. |
| HTTP 401/403 | Zroky or ledger auth failed. | Check API key, bearer token, project id, and ledger scopes. |
| HTTP 404 from ledger | Refund id was not found in the system of record. | Use a valid refund id or adjust the ledger path template. |
| HTTP 404 from CRM | Customer id was not found in the system of record. | Use a valid customer id or adjust the CRM path template. |

## Buyer Demo Script

1. Pick the partner's highest-stakes live-like scenario: refund or CRM/customer record.
2. Run live preflight and show `connector_setup.health_status=healthy` and
   `connector_setup.readiness_status=ready`.
3. Run the full install kit and point to `runtime_policy.allowed=false`.
4. Show `connector_setup.health_status=healthy` and `connector_setup.last_attempts`.
5. Show `outcome_reconciliation.verdict=matched`.
6. Show the 64-character `evidence_hash`.
7. Open the evidence JSON and show decision, policy snapshot, reconciliation,
   audit entries, and hash. Do not show raw secrets.
8. Explain the next live step: connect the partner's real ledger or CRM API and
   repeat the same proof against their system of record.
